# LMAT Runtime Overhead Evaluation

This document describes how we evaluate the runtime cost of using LMAT alongside a live microservice application. It is intentionally method-focused: it explains how we answer the latency and overhead questions without reporting experiment-specific numbers.

## Goal

The objective is to answer reviewer-facing system questions such as:

- What is the P95/P99 latency of the server without LMAT and with LMAT enabled?
- How much throughput is lost when tracing is enabled?
- How much additional overhead comes from LMAT inference beyond tracing alone?

The emphasis is on the monitored application, not on offline model accuracy. We therefore evaluate LMAT as a systems component that competes for CPU, memory, and I/O on the same host as the application.

## Key Comparison Levels

We use three runtime conditions:

| Condition | Purpose |
|---|---|
| `baseline` | Application only, with no tracing and no LMAT |
| `lttng_only` | Application with tracing enabled, but no LMAT inference |
| `lmat_async` | Application with tracing enabled plus co-located LMAT inference replay |

These three levels let us separate:

- application performance with no observability cost
- tracing overhead by itself
- incremental LMAT cost on top of tracing

Because LMAT depends on trace data, the fairest incremental systems comparison is:

- `baseline` vs `lttng_only` for tracing overhead
- `lttng_only` vs `lmat_async` for LMAT overhead

We still report `baseline` vs `lmat_async`, but we interpret that as total end-to-end monitoring cost rather than LMAT-only cost.

## Why We Use Replay-Based LMAT Overhead

Our current VM evaluation does not run a fully live streaming LMAT control loop over the same requests being measured. Instead, we use a replay-based co-location design:

1. Run the application under tracing and collect a real kernel trace.
2. Re-run load on the same host while replaying that real trace through LMAT inference.

This design is appropriate for answering the main systems question:

- how much CPU contention and latency overhead LMAT inference adds when co-located with the application

It is not meant to claim that the model is making real-time tracing decisions on those exact requests. For that reason, we describe `lmat_async` as a co-located inference-overhead proxy, not as a full online deployment.

## Experimental Design

### Target system

We run the Sock Shop microservice benchmark on a single VM and measure end-user request latency under controlled load.

### Workload

The workload is generated by the repository’s load generator and includes realistic Sock Shop actions such as:

- browsing the homepage and catalogue
- viewing item details
- adding and removing cart items
- checkout-related interactions
- login and account-related actions

Each request is logged with:

- timestamp
- scenario name
- success/failure
- end-to-end latency in milliseconds

These request-level logs are the basis for the latency and throughput analysis.

### Fairness controls

To reduce measurement bias, the final overhead protocol includes:

- a warm-up period before each measured condition
- repeated runs at the same load
- rotated condition order across repeats

The rotated order matters because fixed ordering can make later conditions look artificially better due to warmed caches, loaded services, or stabilized background behavior. Rotating the order ensures that `baseline`, `lttng_only`, and `lmat_async` each sometimes run first, middle, and last.

## Measurement Logic

### 1. Baseline

`baseline_load.sh` measures the application without tracing or LMAT. This gives the reference latency and throughput of the system in its least instrumented state.

This condition answers:

- how fast the application is when no monitoring is active

### 2. Tracing only

`lttng_only_run.sh` enables the same tracing infrastructure used by LMAT, but does not run inference.

This condition answers:

- how much overhead is introduced by the observability substrate itself

This is important because tracing is often the dominant systems cost. If tracing already increases tail latency, that cost should not be incorrectly attributed to the model.

### 3. Tracing plus LMAT

`lmat_async_run.sh` first records a real trace, then replays that trace through the trained model while the application serves fresh workload on the same VM.

This condition answers:

- how much extra runtime interference LMAT inference introduces when it shares CPU resources with the application

The async mode is used because it is the most realistic deployment style in this repository: inference runs in the background and does not intentionally block event collection.

## Metrics Reported

For each condition and load level, we report:

- mean throughput in requests per second
- request error rate
- P50 latency
- P95 latency
- P99 latency

These are summarized across repeated runs.

### Why P95 and P99

P95 and P99 are the most relevant metrics for this evaluation because tracing and inference overhead usually show up first in the latency tail rather than in average latency.

### Throughput interpretation

Throughput is interpreted in two ways:

- throughput at the tested operating point
- best observed throughput among the tested load points under an error-rate budget

When only one or two user levels are tested, we describe this conservatively as the best observed throughput among the tested points rather than a strong claim about absolute maximum throughput.

## Output Artifacts

The analyzer writes three main summary files:

- `reviewer_overhead_by_users.csv`
- `reviewer_overhead_max_throughput.csv`
- `reviewer_overhead_summary.md`

These files are intended to support direct reviewer questions about:

- latency without LMAT vs with LMAT enabled
- tracing-only overhead
- throughput under representative load

## How We Interpret the Final Comparison

The final systems interpretation follows this hierarchy:

1. `baseline` establishes the application’s uninstrumented reference point.
2. `lttng_only` quantifies tracing overhead.
3. `lmat_async` quantifies total monitoring overhead when LMAT is present.
4. `lmat_async` minus `lttng_only` is treated as the best estimate of LMAT’s incremental runtime cost.

This keeps the conclusions honest:

- if tracing dominates overhead, we say so
- if LMAT adds little beyond tracing, we say so
- if LMAT adds measurable tail-latency cost, we report that separately from tracing

## Scope and Limitations

This evaluation is designed to answer a systems-overhead question, not a detection-quality question. It tells us how much the host suffers when LMAT is co-located with the application.

It does not by itself measure:

- anomaly-detection quality
- root-cause quality
- correctness of live adaptive trace triggering on the same requests

Those are handled by separate offline change-detection, root-cause, and adaptive-replay experiments elsewhere in the repository.

## Recommended Claim Framing

This methodology supports claims of the form:

- the latency impact of tracing alone
- the total latency impact of tracing plus LMAT
- the incremental overhead of LMAT relative to tracing-only operation
- the throughput sustained under representative load with and without LMAT

This is the intended final evaluation path for reviewer-facing overhead and latency questions.
